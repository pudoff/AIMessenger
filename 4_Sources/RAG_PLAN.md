# План внедрения RAG-ассистента

## Цель

Добавить в проект RAG-ассистента, который отвечает на вопросы пользователя на основе истории сообщений, участников, классификаций и embeddings, уже сохраненных в базе данных.

Основные сценарии:

- Сводка обсуждения в чате.
- Ответ на вопрос по истории переписки.
- Выделение задач, решений, рисков и договоренностей.
- Поиск релевантных сообщений с объяснением ответа.
- Формирование черновика автоответа в стиле пользователя.

## Текущая база проекта

В проекте уже есть важные элементы для RAG:

- `Message` хранит сообщения чатов.
- `MessageClassification` хранит классификацию сообщений.
- `MessageEmbedding` хранит embeddings сообщений.
- `pgvector` используется для семантического поиска.
- `ml-service` умеет строить embeddings через `ml_service.embed_text`.
- Backend уже имеет endpoint `/api/search/semantic/`.
- Celery и Redis уже используются для ML-задач.

Это позволяет строить RAG не с нуля, а поверх существующей инфраструктуры.

## Архитектура MVP

Поток RAG-запроса:

```text
Пользовательский вопрос
-> backend endpoint /api/assistant/rag/
-> проверка авторизации и прав доступа
-> построение embedding вопроса
-> поиск top-N релевантных сообщений через pgvector
-> фильтрация по доступным чатам
-> сбор компактного контекста
-> вызов Qwen или другого OpenAI-compatible LLM
-> возврат ответа и списка источников
```

Важно: RAG должен использовать те же ограничения доступа, что и `/api/search/semantic/`.

## LLM-провайдер

Рекомендуемый подход: сделать универсальный OpenAI-compatible клиент, а не жестко привязываться только к Qwen.

Переменные окружения:

```env
LLM_PROVIDER=qwen
LLM_API_KEY=...
LLM_BASE_URL=https://dashscope-intl.aliyuncs.com/compatible-mode/v1
LLM_MODEL=qwen-plus
LLM_TIMEOUT_SECONDS=30
LLM_MAX_CONTEXT_MESSAGES=15
```

Такой подход позволит позже заменить Qwen на другую OpenAI-compatible модель, локальный Ollama или vLLM.

## Backend-задачи

1. Создать backend-модуль или приложение `assistant`.

Возможная структура:

```text
backend/assistant/
  __init__.py
  apps.py
  serializers.py
  views.py
  services.py
  llm_client.py
  prompts.py
  tests.py
```

2. Добавить endpoint:

```http
POST /api/assistant/rag/
```

Пример запроса:

```json
{
  "query": "Какие задачи по демо еще не закрыты?",
  "chat": 10,
  "limit": 12,
  "mode": "qa"
}
```

Пример ответа:

```json
{
  "answer": "По найденной переписке остались задачи по демонстрации, тестированию и оформлению итогового отчета.",
  "sources": [
    {
      "message_id": 50,
      "chat_id": 10,
      "chat_title": "Команда разработки",
      "sender": "Анна Смирнова",
      "text": "Нужно подготовить демо к пятнице",
      "created_at": "2026-06-04T10:00:00Z",
      "classification": "task",
      "similarity_score": 0.83
    }
  ],
  "model": "qwen-plus",
  "mode": "qa"
}
```

3. Реализовать `RagService`.

Функции сервиса:

- Получить вопрос пользователя.
- Получить embedding вопроса.
- Найти релевантные сообщения.
- Отфильтровать сообщения по правам доступа.
- Собрать prompt.
- Вызвать LLM.
- Вернуть ответ и источники.

4. Реализовать `LLMClient`.

Минимальные методы:

```python
class LLMClient:
    def chat(self, messages, temperature=0.2, max_tokens=800):
        ...
```

5. Добавить prompt-шаблоны.

Базовый system prompt:

```text
Ты AI-ассистент мессенджера "Наш слон".
Отвечай только на основе переданного контекста сообщений.
Если в контексте недостаточно информации, прямо скажи об этом.
Не выдумывай факты.
Пиши кратко и по-русски.
Если выделяешь задачи, указывай исполнителя, срок и источник, если они есть в контексте.
```

6. Добавить режимы:

- `qa` — ответ на вопрос.
- `summary` — сводка обсуждения.
- `tasks` — список задач.
- `decisions` — решения и договоренности.
- `risks` — риски и открытые вопросы.

7. Добавить fallback.

Если LLM недоступна:

- Не падать с 500.
- Вернуть найденные источники.
- Вернуть понятное сообщение: `AI-ответ временно недоступен, но найдены релевантные сообщения`.

## Черновик автоответа

Автоматическую отправку сообщений без подтверждения пользователя в MVP делать не рекомендуется.

Безопасный MVP-вариант: черновик автоответа.

Endpoint:

```http
POST /api/assistant/reply-draft/
```

Пример запроса:

```json
{
  "chat": 10,
  "message_id": 125,
  "instruction": "Ответь кратко и вежливо в моем стиле"
}
```

Пример ответа:

```json
{
  "draft": "Да, посмотрю презентацию сегодня и вернусь с комментариями до вечера.",
  "sources": [
    {
      "message_id": 125,
      "text": "Ты сможешь посмотреть презентацию сегодня?"
    }
  ]
}
```

Стиль пользователя можно строить только по его собственным сообщениям и только в доступных ему чатах.

## Frontend-задачи

1. Доработать страницу AI-ассистента.

Добавить:

- Поле вопроса.
- Выбор режима ответа.
- Выбор чата или режим "по всем доступным чатам".
- Кнопку `Ответить по переписке`.
- Кнопку `Сделать сводку`.
- Кнопку `Выделить задачи`.

2. Показ ответа.

Интерфейс должен отображать:

- Сгенерированный ответ.
- Список источников.
- Название чата.
- Авторов сообщений.
- Даты сообщений.
- Similarity score.
- Кнопку `Показать сообщение`.

3. Черновик автоответа.

В чате рядом с сообщением можно добавить действие:

- `Сформировать ответ`.

Результат не отправлять автоматически, а вставлять в поле ввода как черновик.

## ML-задачи

1. Проверить качество embeddings.

Текущая модель:

```text
sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2
```

Она подходит для MVP на русском языке, но нужно проверить качество на реальных сообщениях.

2. Добавить smoke-тесты RAG retrieval.

Проверить:

- Вопрос про задачи находит сообщения с задачами.
- Вопрос про решения находит сообщения с договоренностями.
- Пользователь не получает сообщения из чужого чата.

3. Продумать reranking.

Для MVP можно оставить pgvector top-N.

На следующем этапе можно добавить reranker:

```text
embedding search -> top 30 -> rerank -> top 10 -> LLM
```

## Безопасность и приватность

Обязательные правила:

- Не отправлять в LLM сообщения из чатов, где пользователь не участник.
- Не логировать полный prompt с приватной перепиской.
- Логировать только request id, user id, chat id, message ids источников, latency и модель.
- Ограничить `limit`, например максимум 20 источников.
- Ограничить длину текста одного сообщения в prompt.
- Добавить timeout для LLM.
- Добавить rate limit на endpoint RAG.

## Тесты

Backend tests:

- Авторизованный пользователь получает RAG-ответ.
- Неавторизованный пользователь получает 401.
- Пользователь не получает источники из чужих чатов.
- `chat` должен быть доступен пользователю.
- Некорректный `limit` возвращает 400.
- При недоступной LLM возвращается fallback.
- Ответ содержит `answer`, `sources`, `model`, `mode`.

Frontend tests/manual checklist:

- AI-ассистент отправляет вопрос.
- Ответ отображается в карточке.
- Источники отображаются под ответом.
- Кнопка `Показать сообщение` открывает нужный чат.
- При ошибке backend показывается понятное сообщение.
- Черновик автоответа вставляется в поле ввода, но не отправляется автоматически.

## Этапы реализации

### Этап 1. RAG MVP

- Backend endpoint `/api/assistant/rag/`.
- Поиск источников через существующий semantic search.
- Вызов Qwen через OpenAI-compatible API.
- Ответ и список источников.
- UI на странице AI-ассистента.

### Этап 2. Режимы анализа

- Сводка чата.
- Список задач.
- Решения и договоренности.
- Риски и открытые вопросы.

### Этап 3. Черновик автоответа

- Endpoint `/api/assistant/reply-draft/`.
- Генерация ответа на выбранное сообщение.
- Вставка результата в поле ввода.
- Пользователь вручную подтверждает отправку.

### Этап 4. Улучшение качества

- Reranking.
- Улучшение prompt-шаблонов.
- Оценка качества на тестовых сценариях.
- Кэширование частых запросов.

## Критерии приемки MVP

- Пользователь может задать вопрос по истории доступных ему чатов.
- Backend возвращает ответ AI и список источников.
- Источники не содержат сообщений из чужих чатов.
- При недоступной LLM приложение не падает.
- Frontend показывает ответ, источники и ошибку при проблемах.
- Сводка и задачи работают хотя бы для выбранного чата.
- Все ключи LLM вынесены в `.env`.
- Полный prompt не пишется в обычные логи.

## Риски

- Галлюцинации LLM.
- Утечка приватных сообщений при неправильной фильтрации.
- Лимиты бесплатного API.
- Большая задержка ответа.
- Низкое качество retrieval на коротких сообщениях.
- Автоответ без подтверждения пользователя может создать репутационные и юридические риски.

## Рекомендация

Для защиты MVP лучше реализовать:

- RAG-вопросы по истории чатов.
- Сводку обсуждения.
- Выделение задач.
- Черновик автоответа.

Автоматическую отправку ответа от имени пользователя лучше не делать в MVP. Безопаснее оставить режим черновика, который пользователь проверяет и отправляет сам.
