# Наш Слон: краткое описание функциональности для QA

Документ нужен как основа для сценариев ручного тестирования и чек-листа. Он описывает, что уже реализовано, где это находится в GUI/API и какой результат ожидается.

## 1. Роли и вход в систему

### Пользователь

GUI:

- `/login` - вход в систему.
- `/register` - регистрация нового пользователя.
- `/forgot-password` - запрос восстановления пароля.
- `/reset-password/:uidb64/:token` - установка нового пароля по ссылке из письма.
- `/app` - рабочее пространство пользователя после входа.

Backend API:

- `POST /api/auth/token/` - получение auth token.
- `POST /api/register/` - регистрация пользователя.
- `GET /api/register/confirm/<uidb64>/<token>/` - подтверждение регистрации по email-ссылке.
- `POST /api/password-reset/` - запрос письма восстановления пароля.
- `POST /api/password-reset/confirm/` - сохранение нового пароля.
- `GET /api/me/` - данные текущего пользователя.

Ожидаемое поведение:

- После регистрации пользователь не должен автоматически попадать в личный кабинет.
- На email отправляется письмо подтверждения регистрации.
- После перехода по ссылке регистрация подтверждается, пользователь возвращается на страницу регистрации/авторизации согласно frontend-настройке.
- После входа token сохраняется на frontend, защищенные разделы становятся доступны.

### Администратор

GUI:

- `/admin` - панель администратора.
- `/admin/users` - управление пользователями.
- `/admin/chats` - корпоративные чаты.
- `/admin/broadcast` - email-рассылка.

Backend API:

- `GET /api/users/` - список пользователей.
- `PATCH /api/users/<id>/` - изменение пользователя.
- `GET /api/admin/events/` - метрики и последние события.
- `POST /api/admin/email/broadcast/` - email-рассылка.

Ожидаемое поведение:

- Пользователь с ролью `user` не должен попадать в `/admin`.
- Пользователь с ролью `admin` должен иметь доступ к административным разделам.

## 2. Основной интерфейс пользователя

GUI пользователя расположен в `/app`.

Основные разделы меню:

- `Чаты` - сводный экран по чатам.
- `Личные сообщения` - личные диалоги.
- `Корпоративные чаты` / групповые чаты - рабочие обсуждения.
- `Контакты` - список контактов и карточка контакта.
- `Сообщества` - раздел сообществ/корпоративных пространств.
- `AI-ассистент` - mock-интерфейс ассистента.
- `Настройки` - пункт меню, если включен в текущей навигации.

Особенности GUI:

- При входе в раздел личных или групповых чатов сначала показывается список доступных чатов.
- После клика по конкретному чату список скрывается, открывается только выбранный чат.
- При возврате к списку последний выбранный чат сохраняется и подсвечивается.
- Stories временно отключены в MVP.

## 3. Контакты

GUI:

- `/app/contacts` - список контактов.
- `/app/contacts/:contactId` - карточка выбранного контакта.

Backend API:

- `GET /api/contacts/` - список контактов текущего пользователя.
- `POST /api/contacts/` - добавить контакт.
- `DELETE /api/contacts/<id>/` - удалить контакт.
- `GET /api/user-search/?q=<строка>` - поиск пользователей.
- `POST /api/contacts/<id>/direct-chat/` - открыть или создать личный чат с контактом.
- `POST /api/contacts/<id>/add-to-chat/` - добавить контакт в групповой/корпоративный чат.

Что проверить:

- Поиск пользователя по имени, логину или email.
- Добавление пользователя в контакты.
- Открытие карточки контакта.
- Наличие ФИО, логина, email, телефона, роли.
- Удаление контакта.
- Кнопка `Написать` открывает или создает личный чат.

## 4. Отправка сообщения

GUI:

- Личный чат: `/app/direct/:chatId`.
- Групповой чат: `/app/groups/:chatId`.

Как отправить:

1. Войти пользователем.
2. Открыть `Личные сообщения` или `Корпоративные чаты`.
3. Выбрать чат из списка.
4. Ввести текст в нижнее поле сообщения.
5. Нажать `Отправить` или клавишу `Enter`.
6. `Shift + Enter` используется для переноса строки.

Backend API:

```http
POST /api/messages/
Authorization: Token <token>
Content-Type: application/json

{
  "chat": 1,
  "text": "Кто отвечает за документацию API?",
  "message_type": "default"
}
```

Ожидаемое поведение:

- Frontend сначала показывает оптимистичное сообщение в ленте.
- Backend сохраняет сообщение и возвращает созданный объект.
- Frontend обновляет сообщения опросом примерно каждые 3 секунды.
- Если отправка не удалась, сообщение помечается ошибкой.
- Пользователь не должен иметь возможность отправить сообщение в чат, где он не участник.

## 5. Классификация сообщения ML-моделью

Когда запускается классификация:

- При создании сообщения через `POST /api/messages/`.
- При изменении текста сообщения через `PATCH /api/messages/<id>/`.

Backend-процесс:

1. `MessageViewSet` сохраняет сообщение.
2. После commit в БД ставятся две Celery-задачи:
   - `classify_message_task(message_id)` - классификация текста.
   - `build_message_embedding_task(message_id)` - построение embedding для semantic search.
3. `classify_message_task` создает или обновляет `MessageClassification`.
4. Статус классификации сначала `pending`.
5. Backend-worker отправляет текст в ML-worker через Celery.
6. ML-worker возвращает класс, confidence и probabilities.
7. Backend сохраняет результат в `MessageClassification`.

Поля классификации в API сообщения:

```json
{
  "classification": {
    "label": "question",
    "confidence": 0.91,
    "probabilities": {
      "question": 0.91,
      "task": 0.05,
      "default": 0.04
    },
    "status": "completed",
    "source": "ml_worker",
    "needs_review": false
  }
}
```

Возможные статусы:

- `pending` - задача создана, результат еще не готов.
- `completed` - классификация завершена.
- `failed` - задачу не удалось поставить или выполнить.

Возможные источники:

- `ml_worker` - результат получен от ML-сервиса.
- `fallback` - применена fallback-логика backend.
- `mock` - тестовый/демо-источник, если используется seed/demo data.

Что должно быть запущено:

- PostgreSQL с pgvector.
- Redis.
- Backend web.
- Backend Celery worker.
- ML worker.

Важное поведение для тестирования:

- Если ML-worker недоступен, сообщение все равно сохраняется.
- Если backend-worker работает, но ML-worker недоступен, backend должен использовать fallback.
- Если backend-worker не запущен, сообщение сохранится, но классификация останется в очереди до запуска worker.
- Если confidence ниже порога `ML_CONFIDENCE_THRESHOLD`, сообщение должно получить `needs_review=true` или label `needs_review`.
- Frontend показывает тег из `message.classification.label`; до готовности классификации может отображаться `default`.

## 6. Embeddings и семантический поиск

Семантический поиск реализован на backend. Отдельного GUI-экрана поиска во frontend на текущий момент не найдено, поэтому проверять нужно через API или Swagger.

Где открыть:

- Swagger/OpenAPI: `/api/docs/`.
- API endpoint: `GET /api/search/semantic/`.

Пример запроса:

```http
GET /api/search/semantic/?q=документация API&limit=20
Authorization: Token <token>
```

Поддерживаемые параметры:

- `q` - поисковый запрос, обязательный.
- `chat` - ID чата, необязательный.
- `date_from` - дата/дата-время начала периода.
- `date_to` - дата/дата-время конца периода.
- `message_type` - `question`, `task` или `default`.
- `limit` - количество результатов, от 1 до 50.

Пример PowerShell:

```powershell
$token = "<token>"
Invoke-RestMethod `
  -Uri "http://127.0.0.1:8000/api/search/semantic/?q=документация API&limit=10" `
  -Headers @{ Authorization = "Token $token" }
```

Ожидаемый ответ:

```json
{
  "count": 2,
  "results": [
    {
      "message_id": 15,
      "chat_id": 3,
      "chat_title": "Команда разработки",
      "sender": {
        "id": 4,
        "username": "ivan"
      },
      "text": "Кто берет на себя документацию API?",
      "message_type": "default",
      "created_at": "2026-05-26T10:00:00+05:00",
      "classification": "question",
      "similarity_score": 0.842311
    }
  ]
}
```

Ожидаемое поведение:

- Пользователь видит результаты только из чатов, где он является участником.
- Администратор проекта может искать по всем чатам.
- Сообщения без embedding не должны ломать поиск.
- Невалидные параметры должны возвращать `400` в едином формате ошибок.
- При недоступности ML embedding для поискового запроса используется fallback embedding.

## 7. Групповые и корпоративные чаты

GUI:

- `/app/groups` - список групповых чатов.
- `/app/groups/:chatId` - выбранный групповой чат.
- `/app/communities` - сообщества/корпоративные пространства.

Backend API:

- `GET /api/chats/?type=group` - групповые чаты.
- `GET /api/chats/?type=corporate` - корпоративные чаты.
- `POST /api/chats/` - создание чата.
- `GET /api/chat-members/?chat=<id>` - участники чата.
- `POST /api/chat-members/` - добавить участника.

Что проверить:

- Создание группового чата.
- Добавление участников из контактов.
- Отображение количества участников.
- Отправка сообщения в группе.
- Пользователь без членства не должен видеть чужой чат.
- Обычный участник не должен управлять участниками, если он не owner/admin чата.

## 8. Административная панель

GUI:

- `/admin` - дашборд.
- `/admin/users` - пользователи.
- `/admin/chats` - корпоративные чаты.
- `/admin/broadcast` - email-рассылка.

Backend API:

- `GET /api/admin/events/` - последние регистрации, созданные чаты, счетчики.
- `GET /api/users/` - пользователи.
- `PATCH /api/users/<id>/` - изменить пользователя.
- `POST /api/admin/email/broadcast/` - отправить рассылку.
- `POST /api/chats/` с `chat_type=corporate` - создать корпоративный чат.

Что проверить:

- Метрики на дашборде загружаются с backend.
- Блок рабочего пространства можно скрыть и показать.
- Список пользователей доступен только администратору.
- Администратор может изменить пользователя.
- Email-рассылка отправляется только администратором.
- Обычный пользователь не имеет доступа к `/admin` и admin API.

## 9. AI-ассистент

GUI:

- `/app/assistant`.

Текущее состояние:

- Реализован frontend mock-интерфейс.
- Ответы формируются на стороне frontend по ключевым словам.
- Backend-интеграция AI-ассистента в этом разделе отдельно не подтверждена.

Что проверить:

- Ввод вопроса.
- Ответы на ключевые слова: `кто отвечает`, `задачи`, `итоги`, `резюме`.
- Кнопки быстрых действий: показать сообщение, напомнить, сформировать итоги недели, показать список задач.

Swagger доступен по адресу:

Что удобно проверять через Swagger:

## 11. Минимальный smoke-сценарий для QA

1. Зарегистрировать пользователя.
2. Подтвердить регистрацию по email.
3. Войти пользователем.
4. Добавить контакт через поиск.
5. Открыть личный чат с контактом.
6. Отправить сообщение с вопросом, например: `Кто отвечает за документацию API?`.
7. Проверить, что сообщение появилось в ленте.
8. Через несколько секунд обновить/дождаться polling и проверить наличие `classification`.
9. Отправить сообщение-задачу, например: `Подготовьте отчет к пятнице`.
10. Проверить, что classification отличается или помечена `needs_review`, если модель не уверена.
11. Выполнить semantic search через `/api/search/semantic/?q=документация API`.
12. Убедиться, что найденное сообщение возвращается в результатах.
13. Войти администратором.
14. Проверить `/admin`, `/admin/users`, `/admin/chats`, `/admin/broadcast`.
15. Проверить, что обычный пользователь не может открыть admin-разделы.

## 12. Что важно вынести в чек-лист

- Раздел GUI или endpoint.
- Предусловия: роль, наличие token, наличие чата, запущены ли workers.
- Действие пользователя.
- Ожидаемый результат в GUI.
- Ожидаемый backend-запрос.
- Ожидаемый ответ API.
- Проверка прав доступа.
- Проверка негативного сценария.
- Фактический результат.
- Статус: пройдено / не пройдено / заблокировано.
